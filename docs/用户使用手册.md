# 宿舍管理系统 — 用户使用手册

> 面向：使用 / 测试本系统的人。
> 前提：已按《环境配置手册》建好 `DormDB` 并能成功启动程序。

---

## 1. 登录

启动程序后弹出登录窗口：

| 输入 | 值 |
|---|---|
| 用户名 | `admin` |
| 密码 | `123456` |

点"登录"进入主界面；密码错误会提示"用户名或密码错误"。

> 安全说明：登录校验走参数化查询（`SqlParameter`），天然防 SQL 注入。

---

## 2. 主界面

顶部菜单栏分四组，状态栏显示当前登录用户：

| 菜单 | 子项 |
|---|---|
| **基础信息** | 宿舍楼管理、班级管理、学生管理、寝室管理 |
| **住宿业务** | 入住登记、调宿、退宿 |
| **查询** | 空床位查询、学生住宿查询、寝室名单 |
| **系统** | 备份与恢复、退出 |

各功能均为独立弹窗，关闭后回到主界面。

---

## 3. 基础信息管理

通用操作：窗口上方表格显示列表，下方填输入框，用"新增/修改/删除/刷新"按钮操作；**点表格某行会把该行内容回填到输入框**，便于修改。

### 3.1 宿舍楼管理
- 字段：楼号（必填、唯一）、楼名、类别（男/女，下拉选择）。
- 删除：该楼下若还有寝室，会提示无法删除（参照完整性）。

### 3.2 班级管理
- 字段：班级名（必填、唯一）、专业（必填）。

### 3.3 学生管理
- 字段：学号（必填、唯一）、姓名（必填）、性别（男/女下拉）、班级（下拉，来自班级表）、电话。
- 删除：该学生若有住宿/记录，会提示需先退宿。

### 3.4 寝室管理
- 字段：楼（下拉）、房号、床位数（1–8）。
- **"新增(含建床位)"**：新增寝室时，会按床位数**自动生成对应数量的床位**（床号 1…N），无需手工建床。
- "改房号"：仅改房号；"删除"：寝室内有人时禁止删除。
- 已住人数 / 状态（空闲/有人/已满）由**数据库触发器自动维护**，无需手动改。

---

## 4. 住宿业务

### 4.1 入住登记
1. 上方表格列出**尚未入住**的学生，点选一名。
2. 下方依次选 **楼 → 房 → 空床**（空床下拉只列当前未被占用的床）。
3. 点"入住"。成功后该生从待入住列表消失，对应寝室人数 +1。
- **约束体现**：同一学生重复入住、或选到已被占的床，会提示"该学生已入住，或该床位已被占用"（一人一寝、一床一人）。

### 4.2 调宿
1. 上方表格列出**在住学生**及其**当前楼/房/床**，点选一名。
2. 下方选目标 **楼 → 房 → 空床**。
3. 点"调宿"。原寝室人数 −1、目标寝室人数 +1。

### 4.3 退宿
1. 上方表格列出在住学生，点选一名。
2. 点"退宿"→ 确认。该寝室人数 −1；**若该寝室退空，状态自动变回"空闲"**（触发器实现）。

> 入住/调宿/退宿都会写入 `HousingLog` 流水表，记录操作人（即登录账号）。

---

## 5. 查询

### 5.1 空床位查询
- 可按"楼"（下拉）和"房号"筛选，留空则查全部。
- 列出所有当前未被占用的床位（楼号/房号/床号）。点"显示全部"清空筛选。

### 5.2 学生住宿查询
- 输入学号或姓名关键字，模糊查询**视图 `v_StudentHousing`**，返回学号/姓名/专业/楼号/房号/床号/入住日期。关键字留空点"显示全部"看全部在住学生。

### 5.3 寝室名单（调用存储过程）
- 选"楼号"+ 填"房号"，点"查询名单"，调用**存储过程 `usp_GetRoomStudents`**，返回该寝室的住宿学生名单（学号/姓名/性别/床号/入住日期）。

## 6. 备份与恢复（系统菜单）

窗口上方是按钮区 + 时间选择器，中间是**操作时间线**（倒序列出每次数据改动：时间/类别/操作/描述/操作人），下方是操作日志文本框。**备份文件固定存放**于 `D:\DataBase\KESHE\backup\`（`DormDB_full.bak`、`DormDB_log.trn`）。

| 按钮 | 作用 |
|---|---|
| 完整备份 | 备份整库（覆盖旧的完整备份；同时清掉脱链的旧日志备份） |
| 日志备份 | 备份事务日志（须先有过完整备份） |
| 刷新日志 | 重新加载操作时间线 |
| 恢复到所选时间点 | 回滚到上方时间选择器所选时刻 |
| 恢复到选中之前 | 回滚到时间线**选中操作的前一条**之后（把选中这条也撤销） |
| 恢复到选中之后 | 回滚到时间线**选中操作**之后（保留这条，撤销其后的） |

### 6.1 演示"基于时间点的恢复（PITR）"的正确顺序 ⭐

新流程不用再手动记时间——靠操作时间线点选即可：

1. 点 **完整备份**。
2. 在系统里**产生数据**（例如给某学生办理入住、或新增一名学生）。
3. 做一次"**误操作**"（例如把刚才那名学生删除）。
4. 点 **日志备份**。
5. 点 **刷新日志**，在操作时间线里**选中那条"误操作"记录**，点 **恢复到选中之前** → 确认。
6. 数据库精确回到误操作之前：误删被撤销，第 2 步的数据找回。

> 说明："之前/之后"都是**整库**回滚到该操作前/后的时刻（不是只撤这一条）；只有当前"完整+日志"备份链覆盖到的时间才能恢复到。

### 6.2 安全提示
- 恢复是**破坏性**操作：会断开该库所有连接并用备份覆盖当前数据，点击前有确认弹窗。
- 程序已内置**失败兜底**：即使恢复中途失败，也会自动把数据库重新拉回在线（恢复到完整备份时刻），**不会把数据库锁死导致无法登录**。
- 若备份报"操作系统拒绝访问"，是 SQL Server 服务账户对备份目录无写权限——见《环境配置手册》第 6 节。

## 8. 异常处理速查

| 现象 | 处理 |
|---|---|
| 登录失败"无法打开数据库 DormDB" | 数据库没建好、连接串不对、或卡在"正在还原"。详见《环境配置手册》第 7 节 |
| 下拉框为空（如班级、楼） | 对应基础表还没有数据，先去基础信息里新增 |
| 入住时空床下拉为空 | 该寝室已住满，或还没建床位 |
| 备份/恢复报错 | 备份目录路径或权限问题，见《环境配置手册》第 6 节 |
| 删除提示"无法删除" | 存在外键引用（楼下有寝室、学生有住宿等），先处理下级数据 |
