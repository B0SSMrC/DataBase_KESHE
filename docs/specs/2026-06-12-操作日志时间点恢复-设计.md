# 设计文档：操作日志驱动的时间点恢复

- 日期：2026-06-12
- 状态：已与用户确认，待转实施计划
- 关联需求：课设要求 (8) 备份/恢复/基于时间点恢复（PITR）的易用性增强

---

## 1. 背景与目标

### 问题
现有 `BackupRestoreForm` 的时间点恢复要求用户**手动记录时间**、再**手填一个 STOPAT 时间**去猜回滚点，易错且不直观（曾因此把数据库卡在"正在还原"）。

### 目标
保持课设要求的**整库 PITR**不变，把"选时间点"做得可视可点：给用户一份**带时间戳的操作清单**，点中某条操作即可定位回滚点，无需手动记时间或猜。

### 明确不做（YAGNI）
- **不做**针对单条记录的"选择性撤销"（只还原某一条、保留其他后续改动）。那是应用层 undo，与 PITR 语义不同，不在本次范围。
- 回滚仍是**整库**回到某一时刻——选中操作之后的所有改动都会一并回退。

---

## 2. 方案选择

采用**方案 1：应用层操作日志表 + 时间戳 STOPAT**。理由：完全复用已实现并加固过的"完整+日志备份 + STOPAT 恢复"引擎，改动小、风险低，且完整满足"看清单 → 点一条 → 回滚"的诉求。

（备选方案 2「标记事务 + STOPBEFOREMARK」更精确但需改造 DBHelper 事务化，改动大，未采用。）

---

## 3. 详细设计

### 3.1 数据库：操作日志表

新增脚本 `sql/10_operation_log.sql`：

```sql
USE DormDB;
GO
CREATE TABLE OperationLog (
    op_id       INT IDENTITY(1,1) PRIMARY KEY,
    op_time     DATETIME2(3) NOT NULL DEFAULT SYSDATETIME(),  -- 毫秒精度，供 STOPAT 使用
    category    NVARCHAR(20) NOT NULL,   -- 宿舍楼/班级/学生/寝室/入住/调宿/退宿
    action      NVARCHAR(10) NOT NULL,   -- 新增/修改/删除
    description NVARCHAR(200) NULL,      -- 友好描述，例 "删除学生 张三(2021001)"
    operator    NVARCHAR(50)  NULL
);
GO
```

> 说明：`OperationLog` 是新增的**通用操作审计日志**；原有 `HousingLog`（住宿流水）保持不变，住宿业务会同时写 HousingLog（原样）与 OperationLog（新增），轻微冗余但互不影响。

### 3.2 记录机制

1. **当前操作员**：登录成功后把操作员名写入静态 `Session.CurrentOperator`（新增极小的 `Session` 静态类），免去给每个窗体加构造参数。
2. **统一写入**：在 `DBHelper` 增加
   ```
   ExecuteLogged(string dataSql, SqlParameter[] dataParams,
                 string category, string action, string description)
   ```
   它在**单个事务**内执行「数据写入 + 插一条 OperationLog（operator 取自 Session）」，保证日志与数据一致、时间戳同步；返回受影响行数。
3. **改造写操作**：以下 7 类写操作把 `DBHelper.Execute(...)` 换成 `ExecuteLogged(...)` 并传中文描述：
   - 宿舍楼 / 班级 / 学生 / 寝室 的**新增、修改、删除**
   - **入住 / 调宿 / 退宿**（住宿三窗体在写 Allocation 的同时记 OperationLog；其原有 HousingLog 写入保留）
   - 寝室"新增含建床位"为多语句操作，整体在一个事务内完成数据 + 日志。

### 3.3 界面：在备份恢复窗体加"操作时间线"

`BackupRestoreForm` 在保留现有「完整备份 / 日志备份 / 时间点恢复 / DateTimePicker / 日志文本框」基础上增加：

- 一个 `DataGridView`，**倒序**显示 OperationLog：列 = 时间 | 类别 | 操作 | 描述 | 操作人。
- 「刷新日志」按钮重新加载。
- **点选某行** → 自动把该行 `op_time` 填入 DateTimePicker。
- 两个语义明确的按钮：
  - **「恢复到该操作之后」** → `STOPAT = 选中行 op_time`（保留该操作及更早的，回退其后的）。
  - **「恢复到该操作之前」** → `STOPAT = 选中行的前一条（更早一条）op_time`（把选中这条及其后都回退）。

### 3.4 回滚机制（复用，不改）

选定 STOPAT 时间后，沿用现有恢复流程，一行不改：
```
SET SINGLE_USER → RESTORE DATABASE full.bak WITH NORECOVERY, REPLACE
                → RESTORE LOG log.trn WITH STOPAT=@t, RECOVERY → SET MULTI_USER
失败 → BringOnline() 自动拉回在线（已实现的兜底）
```
时间线只负责"选对时间"，回滚引擎复用。

### 3.5 新使用流程

| 旧 | 新 |
|---|---|
| 完整备份 → 自己记时间 → 改数据 → 误操作 → 日志备份 → 手填猜时间 → 恢复 | 完整备份 → 改数据/误操作 → 日志备份 → 刷新日志，点中目标操作 → 点"恢复到之前/之后" → 完成 |

## 4. 边界与边界情况

- **选"之前"且选中的是最早一条**：无前一条 → 提示"已是最早记录，如需更早请直接还原完整备份"，不执行。
- **备份链覆盖范围**：只有当前「完整+日志备份」链覆盖到的时间才能恢复到；早于本次完整备份的旧记录虽显示在列表，但恢复不到。界面在时间线上方提示这一点（或标注完整备份基线时间）。
- **时间精度**：`op_time` 取 `SYSDATETIME()`、与数据写入同事务，毫秒级；手工操作间隔通常数秒，STOPAT 取该值不会与相邻操作冲突。
- **OperationLog 随库回滚**：它本身在 DormDB 内，回滚后一并回到该时刻，状态一致、无副作用。
- **日志写入失败**：因与数据同事务，失败则数据一并回滚（保证一致）；向用户提示。

## 6. 验收标准

1. 任一增删改（楼/班级/学生/寝室）及入住/调宿/退宿后，`OperationLog` 新增一条含时间/类别/动作/描述/操作人的记录。
2. 备份恢复窗体能倒序显示该日志并可刷新。
3. 点选一行 + "恢复到之后/之前"，能把库整体回滚到对应时刻：选"之前"误删那条 → 数据找回；选"之后" → 状态符合预期。
4. 恢复失败不卡死（沿用兜底自动上线）。
5. 不破坏现有任何功能；`dotnet build` 通过。
6. `sql/10` 可独立执行；重置库（重跑 01）后流程仍成立。
