# 设计文档：基于快照的"操作撤销"（保留 SQL Server PITR）

- 日期：2026-06-23
- 状态：已与用户确认，待转实施计划
- 背景：SQL Server `BACKUP/RESTORE + STOPAT` 的 PITR 用作"日常撤销"过于脆弱（需管备份文件、依赖日志覆盖、`SINGLE_USER`/`RESTORING` 锁库、还原序列中断遗留 4319 残局）。需要一套稳健、随手可用的"按操作撤销"。

---

## 1. 目标与非目标

### 目标
- 用户每个业务写操作都被记录（已有 `OperationLog`）；在备份窗体的操作时间线选中某操作，一键把数据库**回退到该操作执行之前**的状态。
- 撤销机制**不依赖 SQL Server 备份/恢复**，从根本避免锁库/RESTORING/4319 等问题。

### 非目标（YAGNI）
- 不做"只撤单独一条、保留其后操作"的选择性撤销。撤销语义=**点时间回退**：撤销到操作 X 之前 = 把 X 及其之后的所有操作一起回退。
- 不做 redo。
- 不改动 SQL Server 备份/恢复/PITR（要求 8 的演示项**原样保留**）。

---

## 2. 方案

采用**方案 1：每次业务写操作前给业务数据拍整库快照；撤销=在事务内清空并重载目标快照**。理由：纯应用层 DML，最稳、不锁库、不碰备份文件；天然正确处理多表/建床位/住宿分配等复合操作（整库一致快照）；实现直白、最不易再引入 bug。数据量极小（课设几十行），整表快照代价可忽略。

（已否决：逆操作日志——实现复杂易错；SQL Server DATABASE SNAPSHOT——仍是还原机制、仍会锁库。）

---

## 3. 详细设计

### 3.1 数据库对象（新增 `sql/13_undo_snapshot.sql`）

**影子快照表**（每张业务表一张，列与原表相同 + `snap_op_id INT` 标记属于哪次操作）：
`Snap_Building, Snap_ClassInfo, Snap_Room, Snap_Bed, Snap_Student, Snap_Allocation, Snap_HousingLog`。
（`Admin`/口令、`OperationLog`、`Snap_*` 自身不在快照范围。）

**存储过程 `usp_TakeSnapshot(@op_id)`**：把上述 7 张业务表当前所有行复制进对应 `Snap_*`，打上 `@op_id`。

**存储过程 `usp_RestoreSnapshot(@op_id)`**：一个事务内把业务数据重载为 `@op_id` 这份快照：
1. `DISABLE TRIGGER trg_Allocation_Occupancy ON Allocation`（避免重载 Allocation 时触发器重算覆盖快照值）。
2. 按**子→父**顺序清空：Allocation → HousingLog → Bed → Student → Room → ClassInfo → Building。
3. 按**父→子**顺序重载（每张表 `SET IDENTITY_INSERT ON`，从 `Snap_*` WHERE `snap_op_id=@op_id` 插回，再 `OFF`）：Building → ClassInfo → Room → Student → Bed → Allocation → HousingLog。
4. `ENABLE TRIGGER`。
5. 删除 `OperationLog` 及各 `Snap_*` 中 `op_id/snap_op_id >= @op_id` 的记录（X 及其之后的操作已被撤销）。
6. 全程 `TRY/CATCH`，出错 `ROLLBACK; THROW`。

### 3.2 写入流程（改造 `DBHelper.ExecuteLogged`，仍单事务）

调整事务内顺序为：
1. `INSERT OperationLog(...) OUTPUT INSERTED.op_id` → 取回 `@op_id`。
2. `EXEC usp_TakeSnapshot @op_id`（此刻业务数据 = 该操作执行**前**状态，故 `Snap_*[@op_id]` 即"X 之前"的快照）。
3. 执行数据 SQL（真正的增删改）。
4. 提交。

所有业务写（楼/班级/学生/寝室增删改、建房含建床、入住/调宿/退宿）都经 `ExecuteLogged`，自动获得快照。

### 3.3 撤销流程（界面 → DB）

时间线选中操作 X → 「撤销到选中操作之前」→ 确认弹窗 → `EXEC usp_RestoreSnapshot(X)` → 刷新时间线（X 及其后记录已被该存储过程删除）。纯 DML，不锁库。

### 3.4 自愈

启动时（登录后，紧随 `EnsureOperationLog`）调用 `EnsureUndoInfra()`：幂等创建 7 张 `Snap_*` 表与两个存储过程（proc 用 `EXEC('CREATE OR ALTER PROCEDURE ...')` 动态语句创建以适配 ADO.NET 单批次）。防止 PITR 演示回退掉基础设施后写操作/撤销失效。

### 3.5 界面（`BackupRestoreForm` 重组为两区）

- **上区「数据库备份/恢复（要求 8）」**：完整备份、日志备份、时间点选择 + 「恢复到所选时间点」（SQL Server PITR，保留）。
- **中区「操作时间线」** `DataGridView` + 「刷新」+ **「撤销到选中操作之前」**（新快照撤销）。
- **移除**原基于 PITR 的「恢复到选中之前」「恢复到选中之后」两个按钮（被快照撤销取代，避免两套"之前"混淆）。
- 选中的操作若无对应快照（本功能上线前的旧操作）→ 提示"该操作无快照，不可撤销"。

---

## 4. 已确认决策
- **(a)** 撤销后**删除"该操作及之后"的时间线记录与快照**（撤了即视为未发生），不保留历史、不做 redo。
- **(b)** 快照范围**包含 `HousingLog`**（住宿流水），随业务数据一并回退。
- **(c)** **移除**时间线的两个 PITR 按钮；SQL Server PITR 仅保留「恢复到所选时间点」+ 两个备份按钮。

---

## 5. 边界与风险
- 旧操作（无快照）不可撤销，界面提示。
- 每操作存一份全量业务数据副本；课设数据量极小，不剪枝（如需可后续加"仅保留最近 N 次"）。
- 撤销不记为业务操作、不再次拍快照。
- 快照基础设施在 `DormDB` 内，PITR 可能将其回退掉 → 由启动自愈重建兜底。
- `usp_RestoreSnapshot` 期间临时禁用占用触发器，重载完成后恢复；重载的 Room.occupied_count/status 直接取自快照，保证与 Allocation 一致。

## 7. 验收标准
1. 任一业务写操作后，`Snap_*` 新增一份以该 op_id 标记的整库业务快照。
2. 时间线选中某操作点「撤销到选中操作之前」→ 数据库回到该操作执行前状态（含人数/状态/住宿分配一致），该操作及其之后的时间线记录消失。
3. 撤销过程不锁库、不进入 RESTORING、无需备份文件、可反复进行。
4. SQL Server 备份/恢复/PITR（要求 8）功能保持可用、不受影响。
5. PITR 回退掉快照基础设施后，重启应用能自愈重建，业务写与撤销恢复可用。
6. `dotnet build` 0 错误；不破坏现有任何功能。
